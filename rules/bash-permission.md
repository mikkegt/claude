# Bash 許可プロンプトを埋もれさせない書き方

## なぜ

1つの Bash 呼び出しに複数操作を `&&` / 改行で詰めると、静的解析は
チェーン全体が通らないと承認しない。解析不能トークンが1個混ざるだけで、
無害な `echo` や `ls` ごと承認待ちになり、本当に判断が要る prompt が
不要なものに埋もれる。これを防ぐのが目的。

削減の主役は settings.json の allowlist（読み取り専用操作の恒久許可）側。
ここに書くのは「解析をわざわざ壊さない」最小限のルールに留める。

## MUST

- 詰め込まない。ビルド / スクショ / テスト / `ls` 等の独立操作は別々の Bash
  呼び出しに分ける。分割すれば解析できる大半は自動承認され、確認が要る部分だけ
  prompt に残る
- 静的解析を壊す特定トークンを避ける（これだけで誤検知の大半が消える）:
  - `set -a` / `set +a`（allexport）を使わない。env は `set -a && source .env`
    ではなく、必要な変数だけ `VAR=x cmd` で前置するか、リポジトリに env ロード用
    ラッパーを置いて allowlist する
  - 実行ファイルを変数に入れて `"$VAR"` で起動しない。パスは直書きする
    （例: `"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless …`）
  - 波括弧の中に文字列クォートを置かない（`{"key":val}` は難読化と誤検知される）。
    普通の awk ワンライナー `awk '{print $2}'` は波括弧内にクォートが無いので問題ない
- 作業ディレクトリが既にそのリポジトリなら、先頭の冗長な `cd /abs/path &&` を付けない

## SHOULD

- 波括弧内クォートを避けても解析が通らない複雑なスクリプトに限り、Write で
  ファイル化して実行する（使い捨ての短いワンライナーまでファイル化しなくてよい）

## 経緯

2026-06-16、複数セッション並行作業中に「5秒おきに許可を求められる」相談から制定。
実例の理由文は4種（`set -a` の allexport / 実行ファイルを変数で起動 /
波括弧内クォートの難読化誤検知 / 単なる allowlist 外）で、いずれも
巨大な1コマンドに詰めたため無害な操作ごと承認待ちになっていた。
削減の主役は settings.json allowlist（B）に置き、本ルールは A（書き方）最小限。
背景と継続改善は mikkegt/Jarvis の Issue に集約。
