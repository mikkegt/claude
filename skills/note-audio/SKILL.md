# ノートを音声化する

学習ノート（技術メモ、課題ノート等）のMarkdownを、**物語・読み聞かせ調**のナレーションに書き換えてから Gemini TTS で音声化する。自己啓発書のような単調な朗読ではなく、小説の一節のように耳に入ってくることを目指す。

## 引数

`$ARGUMENTS` にMarkdownファイルのパスを指定する。省略時は以下を表示して終了:

> 音声化したいnote.mdのパスを指定してください。例: `/mikkegt-claude:note-audio path/to/note.md`

## 手順

### 1. Markdownを読み込む

`Read` で対象ファイルを読む。巨大な場合は章単位の分割を提案する。

### 2. 内容を確認する

読み進めながら、理解が曖昧な箇所があれば**ユーザーに質問する**。推測で整形しない。特に:

- 独自略語・固有名詞の読み方
- 文脈が省略されている箇所
- コード例の意図（何を伝えたいのか）

### 3. 物語調ナレーションに整形する

以下の指針で書き換える:

**トーン**
- 小説の地の文のように、場面を描写するつもりで書く
- 「〜です・〜ます」の単調な羅列を避け、接続詞と問いかけで起伏を作る
- 教科書調ではなく、オーディオブックのナレーター口調

**具体的な書き換え**
- 見出しは独立した導入文にする
  - 悪い例: 「3. 構造体」
  - 良い例: 「さて、ここから構造体の話に入ります。少し長い章になりますが、ゆっくり進めていきましょう」
- 箇条書きは接続詞でつなぐ
  - 「まず〜。次に〜。そして最後に〜」
- コード例は情景として描写
  - 悪い例: 「`int x = 5;` と書きます」
  - 良い例: 「例えば、ここで `x` という名前の整数変数を用意して、そこに 5 という値を入れてみます」
- 専門用語は一度だけ原語→読み方→意味を添える
- 章の切れ目で「ひと呼吸おきましょう」のような区切り表現を入れる

**守ること**
- 技術的な正確さを絶対に落とさない
- 情報量を増やすための**創作**はしない（元ノートにない知識は加えない）
- 記号・コード塊は音読可能な形に言い換える（`&&` → 「アンパサンド2つ」等）

### 4. 整形結果をプレビュー

整形した全文をユーザーに見せ、トーンや内容を確認してもらう。必要なら修正。

### 5. TTS実行

整形テキストを一時ファイルに保存し、スクリプトで音声化:

```bash
python3 ~/workspace/github.com/mikkegt/claude/skills/note-audio/tts.py \
  --input <整形テキスト.txt> \
  --output <課題リポ>/docs/audio/ex00.mp3 \
  --title "CPP_Module01_ex00" \
  --album "CPP_Module01" \
  --track 00
```

出力先は原則 `<課題リポ>/docs/audio/<章>.mp3`。課題リポの `.gitignore` に `docs/audio/` が無ければ追記する（mp3はコミット対象外）。

ID3タグ（`--title`, `--album`, `--track`）を付けると、YouTube Music等へアップロード時にアルバム順で連続再生しやすい。アーティストはデフォルトで `JARVIS`。

### 6. 完了報告

生成されたmp3のパスとサイズ、再生確認コマンド（`afplay <path>`）をユーザーに伝える。

## 依存

- `GEMINI_API_KEY` が `~/workspace/github.com/mikkegt/claude/.env` に設定済み
- Python: `google-genai`, `python-dotenv`
- `ffmpeg`（WAV → MP3変換）
