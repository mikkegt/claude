コードをレビューしてください。

## 対象の決定

引数 `$0` から対象を判定する:

| 引数 | 対象 | 取得方法 |
|------|------|----------|
| 数字（例: `123`） | GitHub PRの差分 | `gh pr diff $0` / `gh pr view $0` |
| ファイル/ディレクトリパス | **指定パスのコード全体**（新規/既存問わず） | `Read`（ディレクトリなら配下のソースを再帰的に読む） |
| `staged` | ステージ済みの変更差分 | `git diff --staged` |
| `branch` | 現ブランチの未マージ差分 | `git diff $(git merge-base HEAD main)..HEAD` |
| 空欄 | 作業ツリーの変更差分（未追跡新規ファイル含む） | `git status` で全体把握 → 既存変更は `git diff`、新規ファイルは `Read` |

ポイント:
- **ファイルパス指定はファイル全体レビュー**。生成AIに書かせた新規コードや、既存ファイルを丸ごと見直したい時に使う
- **diff系（`staged` / `branch` / PR番号）は変更箇所のレビュー**。ただし変更ファイルに新規追加分があれば、そのファイル全体も `Read` して文脈を把握する
- **空欄は混在ケースに対応**。`git status` で modified と untracked を両方拾い、modified は diff、untracked は全体を読む
- PRレビュー時は `gh pr view $0` で説明文も取得し、変更意図と実装の整合性も見る

## レビュー6軸

各軸を確認し、判定結果を表にまとめる。

| 軸 | 判定 | 観点 |
|----|------|------|
| セキュリティ | OK / 要修正 | インジェクション、認証漏れ、秘密情報の露出、入力検証 |
| パフォーマンス | OK / 要修正 | N+1クエリ、不要な再レンダリング、重い同期処理、メモリリーク |
| 保守性 | OK / 要修正 | 命名の明確さ、関数長（目安20行）、マジックナンバー、コメントのWHY、構造、非推奨API |
| エッジケース | OK / 要修正 | null/空配列/境界値、並行処理、エラーハンドリング（try/catch・タイムアウト・意味あるメッセージ） |
| 過剰設計 | OK / 要修正 | YAGNI違反、不要な抽象化、使われないオプション、早すぎる最適化 |
| テスト | OK / 要修正 | テストの追加有無、Happy/Edge/Empty/Error/Regression パターン網羅、外部依存のモック |

## 出力フォーマット

1. **概要**: コードが何をしているか1-2文で（差分レビューなら「変更が何をしているか」、ファイル全体なら「ファイルの責務」）
2. **判定表**: 上記の6軸テーブル（要修正の項目は具体的な箇所と改善案を `file:line` 形式で記載）
3. **Good**: 良い点があれば簡潔に
4. **総合判定**: LGTM / 要修正

## 注意

- 推測でコメントしない。実コードを `Read` / `gh pr diff` 等で確認した上で指摘する
- 改善案は具体的に書く（「リファクタすべき」ではなく「`X` を `Y` に分割」のように）
- スタイルの好みではなく、実害のある問題を優先する
